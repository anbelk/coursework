# Решения по evaluation и набору методов

## Целевой pipeline

Фиксируем онлайн-схему:

- retrieval возвращает `top_n = 10` кандидатов;
- LLM оценивает эти 10 кандидатов;
- финальный блок выбирает `top_k = 3` рекомендации.

Причина: `top_n = 5` заметно дешевле, но хуже покрывает три финальные рекомендации. По существующим LLM-оценкам для `author_retriever`:

- при `K = 5`: `llm_has_3_ge2 = 0.725`;
- при `K = 10`: `llm_has_3_ge2 = 0.875`.

То есть top-10 чаще даёт LLM достаточно кандидатов со score >= 2 для заполнения трёх рекомендаций. `top_n = 20` пока не используем как основной режим: для него нет полной LLM-оценки, и он удваивает число LLM-вызовов относительно top-10.

## Целевые метрики

Основные offline proxy-метрики считаем на `@10`, потому что это ровно размер множества, которое дальше уходит в LLM:

- `Hit@10`: есть ли среди 10 кандидатов хотя бы один будущий новый соавтор;
- `MRR@10`: насколько высоко стоит первый будущий новый соавтор;
- `nDCG@10`: насколько хорошо упорядочены все наблюдаемые будущие новые соавторы;
- `Recall@10`: какая доля наблюдаемых будущих новых соавторов попала в top-10.

Целевые LLM-метрики для финального пайплайна:

- `llm_has_3_ge2@10`: хватает ли кандидатов со score >= 2 для трёх финальных рекомендаций;
- `llm_selected3_mean_score@10`: средний score трёх лучших кандидатов после LLM scoring.

Диагностические LLM-метрики:

- `llm_relevant_rate@10_ge2`: доля кандидатов со score >= 2;
- `llm_hit@10_ge3`, `llm_mrr@10_ge3`, `llm_has_3_ge3@10`: строгая проверка по очень сильным кандидатам;
- `llm_ndcg@10`: только служебная диагностика raw retrieval, не метрика финального качества.

Метрики порядка по `score >= 2`, например `llm_mrr@10_ge2`, не используем в основном анализе:
LLM все равно оценивает все 10 кандидатов. MRR имеет смысл только как строгая диагностика по
`score = 3`, где важно, насколько высоко raw retriever ставит первого очень сильного кандидата.

`@5` и `@20` оставляем как диагностические срезы стоимости, но не как основные метрики.

## Скоррелированность сигналов

Да, её можно измерять. Сейчас считаются:

- correlation cosine scores по одним и тем же пользователям и кандидатам;
- Spearman correlation рангов;
- overlap top-K выдач.

Ключевые числа на test:

- `mean_author_embedding` vs `author_retriever`: score Pearson `0.890299`, rank Spearman `0.929669`, top-10 overlap `0.770705`.
- `author_retriever` vs `graphsage_author_metacluster`: score Pearson `0.817162`, rank Spearman `0.813775`, top-10 overlap `0.736797`.
- `graphsage_transformer_author` vs `graphsage_transformer_author_metacluster`: score Pearson `0.899639`, rank Spearman `0.880232`, top-10 overlap `0.641460`.

Это подтверждает, что семантический и графовый сигналы сильно пересекаются. Coauthor graph не даёт полностью независимый источник информации: он во многом кодирует те же тематические сообщества, которые уже видны через embeddings статей.

## Значимость улучшений

Паired bootstrap против `author_retriever` на `@10`:

- `graphsage_transformer_author`, `nDCG@10`: mean diff `0.001294`, 95% CI `[-0.001249, 0.003726]`;
- `graphsage_transformer_author`, `Hit@10`: mean diff `0.002534`, 95% CI `[-0.002534, 0.007603]`;
- `graphsage_author_metacluster`, `nDCG@10`: mean diff `-0.000186`, 95% CI `[-0.001362, 0.000920]`.

То есть улучшения графовых методов на observed coauthor retrieval сейчас не являются статистически устойчивыми на 95% bootstrap-интервале. В тексте работы нельзя писать, что граф значимо улучшает качество. Корректная формулировка: граф даёт сопоставимое качество и иногда улучшает отдельные метрики, но эффект находится в пределах шума разреженного теста.

## Какие методы оставить в основной работе

Основной набор должен проверять разные гипотезы, а не перечислять все похожие варианты:

1. `mean_author_embedding` — простой semantic baseline без обучения.
2. `author_retriever` — обучаемый semantic retriever по статьям автора.
3. `graphsage_author` — GraphSAGE поверх mean author features, то есть проверка добавления coauthor graph к простому semantic baseline.
4. `graphsage_author_metacluster` — проверка мостика clustering -> graph -> recommendation через метакластеры.
5. `graphsage_transformer_author` — объединение transformer author encoder и coauthor graph.

Важно: текущий `graphsage_author` не является чистым graph-only baseline. В реализации начальные
признаки авторов строятся как среднее по embeddings статей до cutoff. Если в работе нужен чистый
графовый бейзлайн, его нужно добавить отдельно, например node2vec/DeepWalk или GraphSAGE с learnable
ID embeddings без paper embeddings.

Не выносить в основной текст как центральные методы:

- `prop_lightgcn_l3_d1p0`, `prop_appnp_s2_a0p1`: хороши по observed coauthor retrieval, но хуже по LLM-качеству;
- fusion-модели: не дали стабильного LLM-прироста;
- `author_retriever_long_p15`: validation стал лучше, test нет;
- `graphsage_transformer_author_metacluster`: метакластерный слой поверх transformer+graph не дал стабильного выигрыша относительно `graphsage_transformer_author`.

## Итоговая позиция

Главная исследовательская история должна быть такой:

1. Семантика статей уже хорошо восстанавливает научное окружение автора.
2. Coauthor graph даёт близкий сигнал, потому что соавторство само формируется вокруг тем, методов и сообществ.
3. Метакластеры полезны как интерпретируемый мост между clustering и recommendation, но пока не дают значимого прироста качества.
4. Комбинирование графа и семантики не приводит к сильному приросту, потому что источники информации скоррелированы.
5. Поэтому финальная система должна быть оценена по консистентной схеме `top_n = 10 -> LLM scoring -> top_k = 3`, а выводы о приростах должны опираться на bootstrap-интервалы, а не на малые разницы в одной метрике.
