# Правила работы над курсовой (AI Coauthor)

Этот файл фиксирует договорённости по стилю и содержанию текста курсовой. Соблюдать при написании
и правке всех глав (`sections/*.tex`).

## Стиль изложения
- Писать как инженер: точно, ёмко, по делу. Без «воды», без образности и красивостей.
- Текст должен быть очеловеченным: допустимы мелкие пунктуационные неровности, не мешающие
  восприятию. Это не значит писать мыльно или расплывчато.
- Минимум кавычек-ёлочек («…»). Люди их часто забывают; не расставлять кавычки там, где смысл ясен
  и без них. Кавычки — только когда действительно цитируем фразу/термин.
- Жаргон поясняй при первом употреблении (в скобках или короткой фразой): мягкая кластеризация,
  когерентность, энтропия назначений и т. п.

## Терминология
- Не переводить коряво устоявшиеся англоязычные ML-термины. Использовать: contrastive loss,
  retrieval, instruct-эмбеддинги, last-token pooling, hard guards и т. п.
- «retrieval» — это этап извлечения кандидатов перед ранжированием. Не путать с задачей целиком.
- Про взаимодополняющую экспертизу (complementary expertise) писать только применительно к этапу
  LLM-переранжирования. На этапе retrieval отбираются ближайшие по вектору — без суждения о
  взаимодополняемости.

## Точность и факты
- Числа и параметры писать так, как реализовано в коде, без «примерно»/«~». Если значение известно
  точно (например, UMAP `n_components = 10`), указывать его.
- Для всего математического приводить формулы (функции потерь, метрики, оценки). Не «забывать»
  формулы.
- Опираться на реальные артефакты (`results/`, `data/`, `models/`). Не выдумывать значения.
- Не оставлять в тексте сомнительных практик как факта. Если метрики прокси — не выдавать их за
  настоящие; сначала посчитать настоящие, потом писать.

## Обоснованность решений
- Каждое архитектурное и числовое решение (выбор модели, веса в формуле, гиперпараметры) —
  обосновывать: что важнее, что менее важно и почему.
- Исключать необоснованные/нерелевантные детали реализации, которые не несут смысла для работы
  (например, выбор SQLite для кэша, формат markdown-ссылок). Если что-то исключено — сообщить
  автору.
- Примеры давать симметрично: если приводятся примеры названий метакластеров, приводить и примеры
  названий обычных кластеров (fine-тем).

## Структура и описание системы
- При описании архитектуры разделять фазы: предподсчёт/обучение (офлайн) и инференс (онлайн).
- Диаграммы рисовать так, чтобы стрелки не накладывались.

## Оформление (LaTeX, правила ВШЭ / ГОСТ Р 7.0.5-2008)
- Следить за правой границей текста: никаких вылетов за поле. Длинные пути/идентификаторы
  оформлять переносимым `\code{}` (рвётся по `/`, `_`, `.`).
- Times New Roman 14, интервал 1,5, красная строка 1 см, выравнивание по ширине.
- Нумерация объектов внутри глав через точку (рис./таблицы/формулы: 2.3). Подписи: рисунки снизу по
  центру, таблицы сверху справа.
- Цитирование и библиография — строго по ГОСТ Р 7.0.5-2008 (biblatex-gost, числовой стиль).
- Сборка: `latexmk -xelatex main.tex` (XeLaTeX + Biber). После правок проверять 0 ошибок и
  отсутствие overfull-вылетов.

## Процесс
- Писать по одной главе, учитывать обратную связь автора и переносить новые правила сюда.
